45 /**
46  * The general idea behind the algorithm:
47  * It uses the granular temperature to determinate if particles in a cell are
48  * mobile enough that the contact forces are worth computation. A cell having a
49  * granular temperature under a value (default is 1e-4) will have an inactive
50  * status which makes them rejected in the broad search step. This results in no
51  * contact forces or velocity integration in future steps for those particles,
52  * which make less computation cost for the simulation.
53  *
54  * Cells may be flagged as that called mobility status:
55  * - mobile (everything is calculated as if feature is not enabled)
56  * - active (particles with low motion next to mobile particles)
57  * - inactive (particles are not in a neighbor’s candidate list of cells around)
58  *
59  * There are some edge cases that need some attention. In these cases, particles
60  * in cells can't be deactivated:
61  *
62  * 1. The solid fraction of the cell is under a value (default = 40%): particles
63  * in the cell may have other forces that can move them and it is supposed that
64  * with this fraction, there is not enough particles around to hold them in
65  * their position.
66  *
67  * 2. Cells having empty cell neighbors: it means that particles are next to a
68  * floating wall/mesh and they may have a change of contact forces from it if
69  * the wall disappears or is moving.
70  *
71  * 3. The cell has cell neighbors which is flagged as mobile from the 3 criteria
72  * mentioned above (granular temperature, solid fractions or floating walls):
73  * this is because motion is badly propagated to particles to the cell around
74  * without this additional "layer" of mobile cells.
75  *
76  * 4. The cell has cell neighbors which is flagged as mobile from the previous
77  * criterion (additional layer of mobile cells): this is again to allow the
78  * propagation of motion, but only the particles in contact with particles
79  * from the mobile cells are considered for the contact force calculation but
80  * their position is not computed at the integration step. Those cells are
81  * flagged as active cells. It works with the assignment and verification of the
82  * mobility status at nodes to check the status of the neighboring cells.
83  *
84  */
85 template <int dim>
86 class DisableContacts

88 public:
89  DisableContacts();

91  /**
92  * Mobility status flag used to identify the status at nodes and the status
93  * of the cell:
94  *
95  * inactive (0)
96  * The movement of particles in the cell is considered as negligible,
97  * particles within this cell are not considered in the contact detection
98  * (rejected at the broad search step), so no force calculation or
99  * integration is applied
100  *
101  * active (1)
102  * The movement of particles in the cell is considered as negligible, but
103  * there's at least one neighbor cell that is flagged as mobile, meaning that
104  * particles need to be in contact candidates lists at the broad search step,
105  * particles directly in contact with the mobile cell are also considered in
106  * force calculation, but none of the particles in the cell are integrated.
107  *
108  * mobile (2)
109  * The movement of particles in the cell is significant or there is at least
110  * one neighbor cell that is mobile by criteria (see the
111  * identify_mobility_status() description), particles need to be in contact
112  * candidates lists at the broad search step and particles are treated as
113  * usual (force calculation and integration)
114  *
115  * empty (3)
116  * This status is only used for the node-based mobility status identification,
117  * no cells are flagged as empty, only node can by identify as empty. Without
118  * this identification of the empty cells, we can't identify the cell that
119  * have a empty neighbor cell, which is critical for simulationd using
120  * floating walls or mesh
121  */
122  enum mobility_status : unsigned int
123  {
124  inactive = 0, // used for cells and nodes
125  active = 1, // used for cells and nodes
126  mobile = 2, // used for cells and nodes
127  empty = 3 // used for nodes only
128  };

144  /**
145  * @brief Carries out the identification of the mobility status of each cell
146  * through a node-based identification and check. Only the active and ghost
147  * cells are processed.
148  *
149  * The following 4 checks (search loops) are done:
150  *
151  * 1. Check if the cell is empty (n_particle = 0), if so, nodes and cells are
152  * flagged as empty mobility status (3)
153  *
154  * 2. Check if the cell is mobile by criteria (average granular temperature >
155  * threshold, solid fraction < threshold or has at least one empty node from
156  * previous check), if so, nodes are flagged and cells are stored with mobile
157  * mobility status (2)
158  *
159  * 3. Check if the cell is mobile by neighbor (at least a node is flagged as
160  * mobile from previous check), if so, cells are stored in map as mobile
161  * status (2) and nodes that are not mobile are flagged as active (1)
162  *
163  * 4. Check if the cell is active (at least a node is flagged as active from
164  * previous check), if so, cells are stored with active status in the map (1)
165  *
166  * The remaining cells are inactive (0)
167  *
168  * @param background_dh The dof handler of the background grid
169  *
170  * @param particle_handler The particle handler that contains all the particles
171  *
172  * @param mpi_communicator The MPI communicator
173  */
