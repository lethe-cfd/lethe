46 /**
47  * @brief Mechanism that disables contacts computation through mobility status
48  * of cells based on granular temperature.
49  *
50  * The general idea behind the algorithm:
51  *
52  * It uses the granular temperature to determinate if particles in a cell are
53  * sufficiently mobile that the contact forces are worth computation.
54  * A cell having a granular temperature under a threshold (default is 1e-4) will
55  * have an inactive or advected status which makes them rejected in the broad
56  * search step. There is also an extra layer of mobile cells around the current
57  * mobile cells to propagate the motion of particles. Additionally, there is
58  * another layer of cells around those mobile cells that are flagged as
59  * static_active or advected_active cells.
60  *
61  * While the inactive status makes the particles static (no further handling),
62  * the advected status will lead to computation and application of a
63  * cell-averaged advected term. This term is computed from the last average
64  * velocity and acceleration of the particles when the cell had a mobile status,
65  * or at the first DEM time step of a CFD iteration in CFD-DEM (ensures an
66  * update of the hydrodynamic forces). Those forces are reapplied after each
67  * DEM time step.
68  *
69  * Particles in cells with an inactive status results in skipping the fine
70  * contact search, the p-p & p-w forces computation and the velocity and
71  * position update. It reduces the computational cost of the simulation when
72  * particles are not moving.
73  *
74  * Particles in cells with an advected status results in skipping the fine
75  * contact search, the p-p & p-w forces computations but they do have a velocity
76  * and position update through the advection term. This is critical for CFD-DEM.
77  *
78  * In summary, cells are flagged using a mobility status:
79  * - mobile (all contacts are calculated, same as if the feature is not enabled)
80  * - static_active (particles with low velocity variance next to mobile
81  * particles)
82  * - advected_active (same as above but particles will be advected)
83  * - inactive (particles next to mobile particles that allow one way contact
84  * forces calculation, not computation of collision pair in the cell)
85  * - advected (same as above, but particles will be advected)
86  *
87  * There are some edge cases that need some attention. In these cases, particles
88  * in cells cannot be deactivated:
89  *
90  * 1. Solid fraction
91  * The solid fraction of the cell is under a value (default = 40%): particles
92  * in the cell are not consolidated (packed) and may move freely.
93  *
94  * 2. Empty cell neighbor
95  * Cell having empty cell neighbor: it means that particles are next to a
96  * floating wall/mesh and they may have a change of contact forces from it if
97  * the wall moves or disappears.
98  *
99  * 3. Additional mobile cell layer
100  * The cell has at least one cell neighbor flagged as mobile from the 3 criteria
101  * mentioned above (granular temperature, solid fractions or floating walls):
102  * this is because motion is badly propagated to particles to the cell around
103  * without this additional "layer" of mobile cells.
104  *
105  * 4. Active cell layer
106  * The cell has at least one cell neighbor flagged as mobile from the previous
107  * criterion (additional layer of mobile cells): again, this is to allow the
108  * motion propagation, but only the particles in contact with particles
109  * from the mobile cells are considered for the contact force calculation but
110  * their position is not computed at the integration step. Those cells are
111  * flagged as active cells. It works with the assignment and verification of the
112  * mobility status at nodes to check the status of the neighboring cells.
113  *
114  * @tparam dim Spatial dimension
115  *
116  */
117 template <int dim>
118 class AdaptiveSparseContacts

120 public:
121  AdaptiveSparseContacts();

123  /**
124  * Mobility status flags used to identify the status at nodes and the status
125  * of the cell:
126  *
127  * inactive (0)
128  * The motion of particles in the cell is considered as negligible,
129  * particles within this cell are not considered in the contact detection
130  * (rejected at the broad search step), so no force calculation or
131  * integration is applied
132  *
133  * static_active (1)
134  * The motion of particles in the cell is considered as negligible, but
135  * there's at least one neighbor cell that is flagged as mobile, meaning that
136  * particles need to be in contact candidates lists at the broad search step,
137  * particles directly in contact with the mobile cell are also considered in
138  * force calculation, but none of the particles in the cell are integrated.
139  *
140  * advected (2)
141  * The variance of the velocity of the particles in cell is low, suggesting
142  * that contact forces are negligible but particles are advected by other
143  * forces. Note that there are currently no mechanisms that distinct active vs
144  * advected status: advection of particles is a user-defined feature. Advected
145  * and active status are detected in the same way, but are not processed the
146  * same at the velocity integration step. For advected particles, the last
147  * computed cell-averaged velocity and acceleration is used to advect the
148  * particles.
149  *
150  * advected_active (3)
151  * Same as (1), but particles are advected by the last computed
152  * cell-averaged velocity and acceleration.
153  *
154  * mobile (4)
155  * The motion of particles in the cell is significant or there is at least
156  * one cell neighbor that is mobile by criteria (see the
157  * identify_mobility_status() description), particles need to be in contact
158  * candidate lists at the broad search step and particles are treated as
159  * usual (force calculation and integration)
160  *
161  * empty (5)
162  * This status is only used for the node-based mobility status identification,
163  * no cells are flagged as empty, only node can be identified as empty.
164  * Without this identification of the empty cells, we cannot identify the cell
165  * that have an empty neighbor cell, which is critical for simulations using
166  * floating walls or mesh.
167  *
168  */
169

174  advected = 2, // used for cells and nodes

176  mobile = 4, // used for cells and nodes
177  empty = 5 // used for nodes only

222  /**
223  * @brief Identify of the mobility status of each cell through a node-based
224  * identification and check. Only the active and ghost cells are processed.
225  *
226  * The following 4 checks (search loops) are done:
227  *
228  * 1. Check if the cell is empty (n_particle = 0), if so, nodes and cells are
229  * flagged as empty mobility status (5)
230  *
231  * 2. Check if the cell is mobile by criteria (average granular temperature >
232  * threshold, solid fraction < threshold or has at least one empty node from
233  * previous check), if so, nodes are flagged and cells are stored with mobile
234  * mobility status (4)
235  *
236  * 3. Check if the cell is mobile by neighbor (at least a node is flagged as
237  * mobile from previous check), if so, cells are stored in map as mobile
238  * status (4) and nodes that are not mobile are flagged as active (1/3)
239  *
240  * 4. Check if the cell is active (at least a node is flagged as active from
241  * previous check), if so, cells are stored with active status in the map
242  * (1/3)
243  *
244  * The remaining cells are inactive (0)
245  *
246  * @param[in] background_dh The dof handler of the background grid.
247  * @param[in] particle_handler The particle handler that contains all the
248  * particles.
249  * @param[in] n_active_cells The number of active cells in triangulation.
250  * @param[in] mpi_communicator The MPI communicator.
251  *
252  */
253  void
254  identify_mobility_status(
255  const DoFHandler<dim> &background_dh,
256  const Particles::ParticleHandler<dim> &particle_handler,
257  const unsigned int n_active_cells,
258  MPI_Comm mpi_communicator);

260  /**
261  * @brief Identify of the mobility status of each cell through a node-based
262  * identification and check. Only the active and ghost cells are processed.
263  * For CFD-DEM, the node status have to be reset at each first DEM iteration
264  * of a CFD time step as mobile in order to update all forces.
265  *
266  * @param[in] background_dh The dof handler of the background grid.
267  * @param[in] particle_handler The particle handler that contains all the
268  * particles.
269  * @param[in] n_active_cells The number of active cells in triangulation.
270  * @param[in] mpi_communicator The MPI communicator.
271  * @param[in] counter The counter of the DEM iteration.
272  *
273  */
274  inline void
275  identify_mobility_status(
276  const DoFHandler<dim> &background_dh,

298  }

301  /**
302  * @brief Map the periodic nodes pairs of the triangulation using the
303  * constraints. It allows to compare the mobility status of the nodes with the
304  * status of the periodic node.
305  *
306  * Note: this might not be the efficient way to pair the periodic nodes.
307  *
308  * @param constraints[in] The background constraints of triangulation.
309  *
310  */
311  inline void
312  map_periodic_nodes(const AffineConstraints<double> &constraints)
313  {
314  periodic_node_ids.clear();
315 
316  IndexSet local_lines = constraints.get_local_lines();
317  for (auto i : local_lines)
318  {
319  for (auto j : local_lines)
320  {
321  // Map the values of the periodic nodes (with repetition)
322  if (constraints.are_identity_constrained(i, j))
323  {
324  periodic_node_ids.insert(std::make_pair(i, j));
325  }
326  }
327  }
328  }

436  /**
437  * @brief Assign the mobility status to the cell, its nodes, and its periodic
438  * coinciding nodes from the periodic nodes map. If required, this will
439  * prevent overwriting a node status prevailing over the initial one.
440  * For instance, empty status must not be overwritten by mobile status because
441  * all cells with this node will be part of the additional mobile layer and
442  * will wrongly propagate mobility status to the next cells.
443  *
444  * @param[in] cell_id The current cell index.
445  * @param[in] dof_indices The vector of the DoF indices of the cell.
446  * @param[in] cell_status The mobility status of cell to assign.
447  * @param[in] node_status The current mobility status of the node.
448  *
449  */
450  inline void
451  assign_mobility_status(unsigned int cell_id,
452  std::vector<types::global_dof_index> &dof_indices,
453  const int cell_status,
454  const int node_status)
455  {
456  cell_mobility_status.insert({cell_id, cell_status});
457 
458  // Assign mobility status to nodes without overwriting empty or mobile nodes
459  // in regard to the case
460  for (auto node_id : dof_indices)
461  {
462  // Prevailing mobility status of the node and assignation
463  int status_assigned = std::max(node_status, mobility_at_nodes(node_id));
464  mobility_at_nodes(node_id) = status_assigned;
465 
466  // Check if node has a periodic node and assign the same mobility status
467  // if prevailing on the one on the periodic node
468  auto it = periodic_node_ids.find(node_id);
469  if (it != periodic_node_ids.end())
470  {
471  mobility_at_nodes(it->second) =
472  std::max(status_assigned, mobility_at_nodes(it->second));
473  }
474  }
475  }
